# MySQL 操作反馈与互斥设计

## 背景

MySQL kit 的连接、切库、选表、翻页和数据修改都会经过浏览器面板、插件 main 与 MySQL 服务三层。当前部分面板已有内部 `busy` 防重，但界面没有同步显示；对象栏与关系图的部分入口还能并发触发。结果是用户点击后短时间内看不到变化，容易连续点击并累积请求。

本次设计面向使用 Harbors 直接浏览和维护 MySQL 数据的开发者。页面的单一任务是：让用户始终知道当前请求正在做什么，并确保同一交互区域不会产生冲突请求。

## 设计原则

1. 请求发出前先渲染忙碌状态，不能等服务端返回后才改变界面。
2. 只锁定会与当前请求冲突的区域，保留无冲突内容的可读性。
3. 服务端确认成功前不改变权威选中态；失败时保留原连接、数据库、表与数据。
4. 每个异步入口都只有一个在途请求；重复点击不产生新请求。
5. 状态文案描述用户动作，例如“正在切换数据库…”，不使用笼统的“处理中…”。
6. 忙碌反馈同时包含可见状态、禁用状态与 `aria-busy` / `aria-live`，并尊重 reduced motion。

## 视觉方向

沿用现有 MySQL 工作台的深海蓝技术界面，不引入新主题或装饰。

- 墨蓝背景 `#07111d`
- 面板蓝黑 `#0a1927`
- 边界蓝灰 `#24445c`
- 正文灰白 `#e8f1f7`
- 活动青色 `#76d0ec`
- 成功绿色 `#6fd2a3`

标题继续使用 `Avenir Next Condensed`，正文使用 `Inter`，数据库名称和状态使用系统等宽字体。新增的唯一视觉签名是 12px 青色环形活动标记：它出现在正在执行的按钮或目标行上，而不是页面中央的大遮罩。

## 交互模型

### 连接栏

- 未连接时只渲染“连接”按钮，不渲染“断开连接”和刷新。
- 已连接时不渲染“连接”按钮，只渲染“断开连接”和刷新。
- 已连接或有请求在途时，连接字段不可编辑。
- 连接、断开、刷新分别显示“连接中…”“断开中…”“刷新中…”，并带活动标记。
- 连接请求发出后立即清空密码；失败后仍保持断开状态和可编辑表单。
- 断开失败继续保留已连接状态；刷新失败继续保留当前对象数据。
- 初次读取连接状态期间禁用表单，状态栏显示“正在读取连接状态…”。

### 数据库对象栏

- 初次读取对象快照时显示轻量加载提示，不误报“未连接”。
- 切换数据库时，目标数据库行显示活动标记，整个对象栏的搜索、数据库和对象按钮暂时不可操作。
- 选择表或视图时，目标对象行显示活动标记；服务端确认后才更新正式选中态。
- 失败时清除活动态、保留原选中项并显示可恢复错误。
- 所有选择入口在已有请求时直接返回，不再使用“后一次覆盖前一次”的方式制造额外请求。

### 数据面板

- 自动加载、翻页、保存和删除使用不同状态文案。
- 请求期间数据内容保持可见，但数据区域覆盖轻量半透明活动层，阻止行选择和分页交互。
- 分页器的页大小、上一页和下一页全部参与禁用；目标页只在请求真正开始时更新。
- 保存期间编辑弹窗保持打开，所有字段、取消和保存控件锁定；按钮显示“添加中…”或“保存中…”。成功后关闭，失败后保留输入。
- 删除期间显示“删除中…”，并禁止再次触发。

### 结构、关系图与 SQL

- 结构面板没有用户发起的远端操作，保留现有加载文案并补充 `aria-busy`。
- 关系图加载与重试共用单一在途状态。点击重试后立即清除错误、进入加载态，重复点击不发请求。
- 打开关系图中的表时锁定关系图点击并显示“正在打开 <表名>…”，失败后留在关系图并显示错误。
- SQL 执行期间禁用编辑器，执行按钮显示活动标记和“执行中…”，状态栏显示“正在执行 SQL…”。结果仍对应界面中不可变的那条 SQL。

## 状态与并发

每个面板维护语义化的本地活动状态，而不是只有布尔值：

```ts
type ConnectionActivity = 'hydrate' | 'connect' | 'disconnect' | 'refresh' | null;
type ExplorerActivity = { kind: 'hydrate' | 'database' | 'object'; name?: string } | null;
type DataActivity = 'load' | 'page' | 'save' | 'delete' | null;
type RelationshipActivity = { kind: 'load' | 'open'; name?: string } | null;
```

活动状态必须在调用 `message.request` 前写入并渲染，在当前请求完成、失败或面板卸载时清除。现有 revision / sequence 校验继续负责拒绝旧响应；活动状态负责阻止新的冲突请求。两者不能互相替代。

## 错误处理

- 错误信息出现在原有错误槽位，不覆盖仍可使用的数据。
- 连接、切库、选表、保存、删除与打开关系表失败后恢复操作能力。
- 迟到响应或迟到错误不得清除新挂载面板的活动状态。
- 错误状态不自动重试；用户明确点击重试或重新执行。

## 验证标准

1. 每个异步操作的测试使用未决 Promise，断言 Promise 完成前已经出现活动文案或标记。
2. 未决期间连续点击只产生一次对应请求。
3. 连接与断开按钮在任意稳定状态下严格二选一。
4. 失败测试证明旧连接、旧选中项、旧数据或弹窗输入得到保留。
5. MySQL kit 全量测试和仓库 `npm run check` 通过。
6. 单 kit 页面用真实服务器验证连接、切库、选表、翻页和 SQL 执行的即时反馈。
