# 提交信息规范与本地历史整理设计

## 目标

恢复 Harbors 已有的方括号提交类型格式，形成可查阅、可被 Codex 执行的仓库规范，并修正尚未推送的本地提交信息。历史整理只改变提交消息和由此产生的提交 SHA，不改变任何提交对应的文件树。

## 提交信息规范

提交标题必须匹配：

```text
^\[(Init|Feature|Bug|Optimize)\] .+
```

类型含义：

| 类型 | 使用范围 |
| --- | --- |
| `[Init]` | 仅用于仓库初始化 |
| `[Feature]` | 新功能、文档、测试以及面向使用者或维护者的新增内容 |
| `[Bug]` | 修复错误、回归或不符合预期的行为 |
| `[Optimize]` | 不改变预期功能的重构、性能优化、结构调整和维护性改进 |

类型大小写固定。摘要使用简洁中文，不以句号结尾。一个提交只表达一个可审查的逻辑改动。

## 规范落点

规范同时落在两个层次：

- `docs/guides/development-workflow.md`：作为面向开发者的完整说明，包含类型、格式、示例和提交前检查。
- 仓库根 `AGENTS.md`：作为 Codex 的持久仓库指令，引用开发工作流并直接列出必须遵守的格式和类型选择。

功能开发 Skill 中现有的“使用简洁提交消息”改为明确遵循本规范，防止自动化流程继续产生“功能：”“文档：”等新格式。

## 历史整理范围

只整理没有进入 `origin/main` 的本地历史：

- `main` 相对 `origin/main` 的 7 个现有提交。
- `codex/feature-workflow-skill` 相对 `main` 的 1 个提交。
- `feat/database-kits-zh-cn` 指向上述本地历史中的数据库中文化提交，需要同步指向对应的新提交。
- 本设计和后续实施产生的新提交从创建时就使用规范格式，无需二次改名。

以下已经进入 `origin/main` 的 3 个历史例外保持不变：

- `c730e3f [optimize] 优化 tsc 编译速度`
- `477e8a1 重构：迁移并加固多工作区架构`
- `b362012 文档：重建项目知识库与架构治理说明`

不执行任何远端 push 或 force push。

## 消息转换

本地提交按以下确定性规则转换：

- `功能：<摘要>` → `[Feature] <摘要>`
- `文档：<摘要>` → `[Feature] <摘要>`
- `修复：<摘要>` → `[Bug] <摘要>`
- `重构：<摘要>` 或 `优化：<摘要>` → `[Optimize] <摘要>`
- 已符合规范的标题保持不变。

转换只处理标题，不改写提交正文。若出现无法匹配的非规范标题，流程停止并报告，不猜测类型。

## 安全执行方案

1. 确认三个 worktree 均干净，并记录所有相关分支旧 SHA、提交数量和 tree SHA。
2. 使用 `git bundle` 在 `mktemp -d` 创建的临时目录保存 `main`、`feat/database-kits-zh-cn` 和 `codex/feature-workflow-skill`，作为可恢复备份。
3. 验证本地待整理历史为线性历史且不包含 merge commit。
4. 在 Git 对象层按顺序重建 `main` 的本地提交：保留每个提交的 tree、作者、作者日期、提交者和提交者日期，只转换消息并映射到新的父提交。
5. 让 `feat/database-kits-zh-cn` 指向数据库中文化提交对应的新 SHA。
6. 将 `codex/feature-workflow-skill` 的单个功能提交重放到新的 `main`，保留其完整 diff，并把标题转换为规范格式；同步加入 Skill 的提交规范说明后保持为一个提交。
7. 使用带旧 SHA 前置条件的 `git update-ref` 更新三个本地分支，避免并发变化被覆盖。
8. 保留所有 worktree，不执行删除、hard reset 或远端写入。

## 验证

历史整理完成后必须验证：

1. `origin/main` SHA 完全不变。
2. 三个 worktree 均保持干净，分支名称不变。
3. `main`、数据库分支和功能 Skill 分支的最终文件树与规范文档修改之外的预期内容一致。
4. 被整理范围内的所有提交标题都匹配规范正则；3 个远端历史例外是唯一保留的异常。
5. 本地 main 的提交数量和相对顺序不变；功能 Skill 分支仍只比 main 多一个提交。
6. `git diff --check`、Skill validator、`npm run test:feature-workflow` 和 `npm run check` 全部通过。

## 恢复策略

如果任一 tree、分支位置或验证结果不符合预期，立即停止。使用 bundle 中记录的旧 SHA 恢复本地 refs；不在错误状态上继续追加修复，也不接触远端历史。
