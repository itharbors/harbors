# SQLite 与 MySQL 关系图布局优化设计

## 背景

SQLite 与 MySQL 的 Relationships Panel 已能展示全库表、字段和外键，但两个插件分别维护一份近似相同的关系图实现。当前算法以外键方向分层，同层节点从上到下按名称排列，无外键的表固定为三列；布局不接收实际画布尺寸，也不允许拖动节点。Panel 只在内存中保存缩放和平移，重新打开数据库后不会恢复上次视图。

本次优化把关系图明确为可持久化的图编辑视图：相近业务名称的表尽量靠近，外键关系仍影响组内位置，用户可以拖动节点微调；自动排列和重置排列依据当前可见区域重新计算，而不是使用固定列数。

## 目标

- SQLite 与 MySQL 使用同一套确定性布局、拖动交互、缓存格式和冲突处理规则。
- 从表名提取稳定、可解释的业务相似度，使相近功能的表优先进入同一视觉分组。
- 自动排列同时尊重外键关系、名称分组、节点实际高度和当前画布宽高。
- 每个数据库拥有稳定的布局身份；重新打开时恢复节点位置、缩放和平移。
- 数据库结构变化后保留仍存在表的位置，移除失效表，并把新增表放入合适分组。
- 缓存冲突、损坏、版本不兼容或写入失败不能阻止关系图显示和操作。
- 保留现有搜索、键盘打开表、关系明细、缩放、平移、适应窗口和无障碍行为。

## 非目标

- 不把布局写入 SQLite/MySQL 数据库，不创建元数据表，也不要求写权限。
- 不在不同设备或用户之间同步布局。
- 不通过模型或远程服务理解表名语义。
- 不改变 Schema Panel、数据库连接方式、外键读取方式或业务数据。
- 不把名称相似度渲染成用户可见的虚拟连线。

## 方案选择

### 方案一：延续外键分层，仅改排序和列数

实现风险最低，但名称只能作为同层排序条件，不能真正形成业务分组；没有外键的表仍与关系图割裂。视口自适应也只能修补固定分层的结果。

### 方案二：外键图与名称相似度结合的分组布局

外键作为结构边，名称相似度作为仅供布局使用的加权边。算法先形成业务分组，再在组内安排节点并按视口纵横比打包各组。结果确定、可测试、无需新运行时依赖，也能自然兼容手工拖动和缓存。本设计采用此方案。

### 方案三：引入 ELK、Dagre 或通用力导向引擎

通用引擎能处理复杂图，但会增加依赖和适配层；力导向结果容易随输入和迭代参数漂移，难以保证重置布局的稳定性。名称分组、变高节点、缓存合并和现有边路由仍需自行实现，收益不足以抵消复杂度。

## 架构

新增浏览器安全的 workspace 包 `packages/relationship-graph`，不得导入 Node API、数据库驱动、DOM 私有宿主对象或 SQLite/MySQL 插件实现。它提供：

- 通用图、节点位置、视口和持久化记录类型；
- 表名分词、名称相似度和分组算法；
- 接收画布尺寸的自动布局与边路由；
- 缓存身份规范化、键生成、读取、校验、冲突桶和写入；
- Schema 变化时的缓存协调；
- 通用关系图 DOM 渲染和拖动、缩放、平移控制器。

SQLite 与 MySQL Relationships Panel 只负责：

- 从各自 core 获取连接状态和关系图；
- 把连接状态转换成数据库身份描述；
- 提供数据库类型、节点类型标签、ARIA 文案和打开表回调；
- 在连接或 Schema 事件到来时调用共享控制器；
- 使用各自主题 CSS 渲染共享 class 结构。

共享包使用自身的通用关系图类型，不依赖 `sqlite-contracts` 或 `mysql-contracts`。Panel 在边界处做结构兼容的显式映射，避免共享包反向耦合数据库协议。

## 数据库身份与缓存键

### 身份描述

缓存身份是可核验的规范化字符串，而不是仅凭显示名称推断。

- SQLite：`sqlite\0<canonical-real-path>\0<file-identity>`。core 在打开文件后提供规范化真实路径和只读文件身份。文件身份优先使用 `dev:ino`；平台无法提供有意义的 inode 时使用创建时间，最终退化为规范化真实路径。替换同一路径下的数据库文件时，只要文件系统能识别为不同文件，就使用新的布局身份。
- MySQL：`mysql\0<normalized-endpoint>\0<database>`。endpoint 使用连接层已经确定的主机和端口规范形式，database 使用服务器返回或已验证的库名。不同账号访问同一服务中的同一库共享布局，因为它们展示的是同一逻辑 Schema。

身份描述不得包含 MySQL 密码、连接令牌或其他凭据。未选定 MySQL database 时不读写布局缓存。

### 缓存键和冲突桶

共享包用 Web Crypto SHA-256 计算身份描述摘要，缓存键格式为：

`itharbors:relationship-layout:v1:<engine>:<sha256>`

键中的引擎命名空间阻止 SQLite/MySQL 交叉命中。值不是裸布局，而是带完整身份的冲突桶：

```ts
type LayoutBucketV1 = {
  version: 1;
  entries: Array<{
    identity: string;
    updatedAt: number;
    state: PersistedRelationshipStateV1;
  }>;
};
```

读取时必须在桶内逐字匹配完整 `identity`，不能只相信摘要。摘要碰撞时，不同身份作为独立 entry 共存；同一身份只更新自己的 entry。桶解析失败、结构校验失败或没有精确身份时视为缓存未命中。每个桶设置有限 entry 数，超限时只淘汰该桶中最久未更新的记录，不扫描或删除其他应用数据。

缓存状态包含版本、节点坐标、视口和上次可见画布尺寸：

```ts
type PersistedRelationshipStateV1 = {
  nodes: Record<string, { x: number; y: number }>;
  viewport: { x: number; y: number; scale: number };
  canvas: { width: number; height: number };
};
```

表名作为该数据库身份内的节点键。所有数字必须有限且处于合理边界；未知字段被忽略。localStorage 不可用、配额不足或写入抛错时，控制器保留当前内存状态并停止本次写入，不向用户显示阻断性错误。

## 表名相似度与分组

### 规范化和分词

名称先按 Unicode 小写规范化，再按下划线、连字符、空白、点号、camelCase 边界和数字边界拆词。空 token 被移除；英语单复数只做保守归一化，例如 `users` 与 `user` 可视为同词，但不进行词典式词干推断。

常见技术性尾词如 `map`、`link`、`rel`、`history`、`log` 和 `detail` 不会被丢弃，但权重低于首个业务 token。这样 `user_profile`、`user_role`、`users` 的相似度高于 `audit_log`，同时仍保持完全确定且无需语言模型。

### 加权布局边

布局图包含两类边：

- 外键边：高权重、用户可见，决定强结构邻接和最终连线路由。
- 名称边：仅布局可见。共享首个规范 token、共享多个 token、长公共前缀或自然名称相邻会分别增加权重。

为避免所有带 `id`、`data` 等通用 token 的表聚成一个巨型分组，只在相似度超过阈值时建立名称边，并限制每个节点保留分数最高且稳定排序后的少量名称邻居。并列时按完整自然名称排序，保证相同输入产生相同输出。

外键连通分量不会强制吞并全部业务组。算法以加权边形成小型社区：外键保证紧邻关系，名称边帮助没有外键或跨模块引用较少的表靠近同类表。分组键取组内自然排序最小的规范名称，用于稳定排序。

## 自动布局

### 输入和确定性

布局函数接收关系图以及 `{ width, height }` 的当前画布尺寸。尺寸会被钳制到安全的正数；测试或隐藏 Panel 无法获得尺寸时使用 `960 × 640`。相同图、相同尺寸和相同配置必须得到逐值相同的节点位置与边路径。

节点宽度保持统一，节点高度继续由字段行数决定。算法不使用随机数、当前时间或遍历不稳定的 Map/对象顺序。

### 两级排列

1. 计算名称 token、加权边和稳定业务分组。
2. 组内先按照外键方向和强邻接生成局部层次；循环外键通过强连通分量压缩处理。
3. 没有足够外键方向的节点按自然名称顺序填充局部网格，相似名称连续放置。
4. 为每个分组计算包含节点间距和连线路由余量的包围盒。
5. 根据当前画布纵横比试算候选列数，以整体包围盒与画布纵横比差、空白面积和组间连线长度的加权成本选择结果。
6. 以稳定分组顺序进行 shelf packing；窄画布倾向单列或少列，宽画布使用更多列，节点不会重叠。
7. 在最终节点位置确定后路由外键。自引用、平行约束、同列关系和跨组关系继续拥有独立可读路径，所有路径坐标都计入最终布局边界。

自动布局完成后计算适应窗口视口。最小和最大缩放边界沿用当前行为，避免大库被缩成不可操作的小点或无限放大。

## 缓存恢复与 Schema 协调

连接并取得关系图后按以下次序建立视图：

1. 计算数据库身份并读取精确匹配的缓存 entry。
2. 没有有效缓存时，使用当前画布尺寸自动布局并适应窗口。
3. 有缓存时，仅恢复当前 Schema 中仍存在且坐标有效的节点。
4. 已删除表的位置被丢弃。
5. 新增表先按当前图计算其业务分组，再放入同组节点包围盒附近的第一个无碰撞位置；若新增数量较多或无法无碰撞安放，则对受影响分组局部重排，不移动无关分组。
6. 恢复并钳制缓存视口；若所有节点均在视口外或状态明显不可用，则改为适应窗口。
7. 协调后的完整状态回写同一身份 entry。

Schema 事件不会覆盖用户手工位置。只有用户点击“自动排列”才全量放弃旧节点坐标。

## 交互

### 节点拖动

表卡片可用指针拖动。按下卡片时捕获指针并阻止画布平移；移动过程通过 `requestAnimationFrame` 更新节点和相连边，拖动结束后提交位置。点击或键盘 Enter/Space 仍打开 Schema Panel；只有超过小幅移动阈值才视为拖动，避免点击误触。

拖动位置不强制吸附网格，但会钳制为有限坐标。节点允许被移到当前视口之外，画布仍可平移找到它。后续“适应窗口”会包含全部节点。

### 画布与工具栏

- 搜索：行为不变，只高亮/弱化节点，不改变布局。
- 缩放、滚轮缩放、画布平移：行为不变，交互结束后保存视口。
- 适应窗口：只根据现有节点包围盒更新视口，不移动节点。
- 自动排列：读取按钮触发时的实际画布尺寸，全量重新计算节点和边，随后适应窗口并覆盖当前数据库缓存。

`ResizeObserver` 只记录最新可用尺寸，不在每次窗口变化时自动重排，避免用户手工布局被窗口缩放破坏。自动排列按钮在加载或打开表的活动状态中禁用，与现有 MySQL 交互规则一致。

持久化采用短防抖：节点拖动结束立即安排保存，连续缩放和平移合并写入。Panel unmount 前执行一次同步 best-effort flush；异步身份摘要尚未就绪时保留待写状态，摘要完成后再写入，但不得在已切换数据库后写到旧身份。

## 状态与并发边界

控制器为每次数据库连接分配递增 generation。身份计算、关系图加载和缓存读取都携带 generation；数据库切换、断开或 Panel unmount 后的旧结果被丢弃。

同一数据库可能在多个 Harbors 窗口打开。写入 entry 带 `updatedAt`，每次保存前重新读取桶并只替换本身份 entry，避免覆盖同桶其他身份。对于同一身份的并发窗口采用最后完成写入者获胜；`storage` 事件不实时搬动另一个已打开窗口中的节点，避免远端布局突然覆盖正在拖动的本地状态。下次重新打开时读取最新记录。

## 错误处理

- 身份缺失：继续使用内存布局，不缓存。
- Web Crypto 不可用：使用共享包内稳定的非加密摘要作为键，但仍依赖桶内完整身份匹配消除误命中。
- 缓存 JSON 损坏或类型无效：忽略该桶并自动排列；下一次成功保存会替换为合法结构。
- localStorage 读取或写入失败：忽略持久化错误，关系图保持可用。
- 自动布局失败或得到非有限坐标：回退到按自然名称、视口自适应列数的简单网格。
- 单个异常外键引用不存在的表：忽略该布局边，关系图中的有效表仍显示。

缓存异常不复用数据库加载错误 UI，因为它不影响核心数据读取；开发日志可以记录一次诊断信息，不能包含凭据或完整 MySQL 连接输入。

## 性能

名称边通过 token 倒排索引生成候选，禁止对全部表做无界 O(n²) 两两比较。每个节点的名称邻居数有固定上限。强连通分量和外键遍历保持迭代实现，继续支持五千表链而不耗尽调用栈。

拖动只更新被移动节点及其关联边；DOM 节点不在每个 pointermove 中整体重建。缓存保存只序列化当前数据库的节点位置，不保存字段和外键内容。

## 测试策略

### 共享包单元测试

- snake_case、camelCase、数字、Unicode、单复数和技术尾词分词。
- 名称相近表进入相邻分组，不相关表不会因通用 token 聚成一组。
- 外键、循环、自引用、平行边、孤立表和变高节点布局无重叠且确定。
- 同一图在宽、窄、纵向视口中选择不同打包列数，并更接近对应纵横比。
- 五千表链不递归溢出；大批孤立表不执行全量两两比较。
- 缓存键命名空间、完整身份核验、摘要碰撞桶、损坏数据、非法坐标、配额错误和版本不兼容。
- Schema 增删表时保留旧位置、丢弃删除项并把新增项放到相似分组附近。

### Panel 行为测试

SQLite 与 MySQL 分别覆盖：

- 首次打开自动排列，重新打开同一数据库恢复缓存。
- 不同 SQLite 文件、不同 MySQL endpoint/database 不共享布局。
- 拖动节点后边路径更新，释放后保存；小幅点击仍打开 Schema Panel。
- 适应窗口不改变节点坐标；自动排列读取当前 canvas 尺寸并覆盖缓存。
- 数据事件不重新加载或重排图，Schema 事件执行增量协调。
- 切换数据库时迟到的身份、缓存或关系图 Promise 不污染新连接。
- localStorage 和 Web Crypto 异常不会显示数据库加载失败。
- 搜索、键盘、ARIA、关系明细、缩放和平移现有回归测试继续通过。

### 集成与构建验证

- 构建并检查 `packages/relationship-graph`、SQLite Relationships 插件和 MySQL Relationships 插件。
- 运行 `npm run test -w @itharbors/kit-sqlite` 与 `npm run test -w @itharbors/kit-mysql`。
- 运行根级构建和与共享包接入相关的仓库测试。
- 分别启动 SQLite 和 MySQL Kit，人工验证宽/窄窗口、拖动、重开恢复、适应窗口、自动排列、Schema 增删表和数据库切换。

## 实施边界

实现按测试驱动分为共享协议与缓存、名称分组与视口布局、通用交互控制器、SQLite 接入、MySQL 接入以及回归验证。不会在两个插件中分别复制新算法。已有视觉主题继续由各自 CSS 控制，共享包只约定稳定 class 和行为接口。
