# 数据库 Kit 轻量产品优化设计

## 背景

基于最新 `origin/main` 对 SQLite、MySQL 两个 Kit 的首屏、连接入口、空状态、现有产品规格和聚焦测试进行走查。两套 Kit 的核心工作流、布局和异步反馈已经完整，本次只修正三个不会改变能力边界的小问题：SQLite 首次文件浏览目录不符合用户预期、SQLite 结构页的前置状态提示不够准确，以及 MySQL 必填连接参数会把英文协议校验错误直接暴露到中文界面。

## 目标与边界

1. SQLite 没有最近数据库时，文件浏览器从当前用户目录开始；已有最近数据库时仍从最近文件所在目录开始。
2. SQLite 结构页分别表达“未连接”和“已连接但未选择对象”，不把两个状态合并成同一句提示。
3. MySQL 在发起连接请求前校验主机、端口和用户名，并在连接栏显示中文错误；database 继续保持可选。

不改变 Kit 布局、数据库协议、连接安全策略、SQLite 默认只读行为、MySQL 可选 database 流程，也不新增两套 Kit 的功能对齐或视觉重做。

## 方案

### SQLite 文件浏览起点

SQLite core 增加只返回 `os.homedir()` 的 `getDefaultDirectory` 请求。connection Panel 打开文件浏览器时先读取最近数据库；有记录时沿用最近文件目录，没有记录时请求默认目录，再调用既有 `listDirectory`。目录枚举、文件过滤、最近记录和错误处理保持不变。

默认目录由 Node 侧提供，避免浏览器 Panel 猜测操作系统路径，也避免把 `.` 解释为服务进程工作目录。若默认目录不可访问，既有文件浏览错误仍在连接栏展示，用户可以重试。

### SQLite 结构空状态

结构 Panel 按顺序渲染：请求错误、未连接、未选择对象、加载中、已加载。未连接提示用户先打开数据库；已连接但未选择对象提示从资源管理器选择对象。状态栏同步使用“等待数据库连接”和“等待选择数据库对象”。

### MySQL 连接参数校验

connection Panel 在进入异步 `connect` 前执行本地校验：主机和用户名去除首尾空白后必须非空；端口必须是 1–65535 的整数。失败时不调用 core、不清空密码、不进入忙碌态，连接栏用中文错误说明具体字段，并把焦点移到第一个无效输入。

输入仍由 core 完整复核，Panel 校验只是即时产品反馈，不能替代服务端安全边界。database 为空时继续发送 `null`，密码和 TLS 语义不变。

## 测试与验证

- SQLite core / connection Panel 测试覆盖默认用户目录和最近数据库优先级。
- SQLite schema Panel 测试覆盖未连接与未选择对象的独立提示和状态栏。
- MySQL connection Panel 测试覆盖空主机、非法端口、空用户名不发请求，以及合法的无 database 连接仍可执行。
- 构建并检查受影响插件，运行 SQLite、MySQL Kit 全量测试，再运行仓库完整检查。
- 在最新单 Kit 页面复查两套首屏；SQLite 文件选择交互若浏览器自动化无法操作 sandbox iframe，则以真实 Panel DOM 测试和运行时测试作为交互证据，并明确记录限制。
